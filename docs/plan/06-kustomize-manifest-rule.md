# 06 — Kustomize Manifest Rule

The current `k8s_service_manifest` hardcodes `k8s/overlays/<env>/` and
`k8s/base/`. Replace with a rule that takes explicit file targets and
computes paths from `ctx.label.package` or caller-provided attributes.

## Prerequisites
- 04, 05 complete.

## Inputs
- Reference: `pave-infra-monorepo/build_defs/k8s/service_manifest.bzl`.
- PRS §3 ("Parameterized Manifest Pathing").

## Outputs

### File: `lifecycle/manifests/kustomize_manifest.bzl`
Exposes a public **rule** (not macro — see note) `kustomize_manifest`.

**Note on rule-vs-macro:** the old code was a macro wrapping a private rule
because it needed `native.glob(["k8s/**"])`. The new rule has no implicit
globbing; callers pass explicit `base_manifests`, `overlay_manifests`, and
`kustomization` files. A thin `kustomize_manifest` **macro** wrapping the
rule is still provided for ergonomic defaults.

### Rule: `_kustomize_manifest`
Attrs:
- `base_manifests` — `label_list(allow_files = [".yaml", ".yml"])`. Any
  resources referenced by the overlay's `kustomization.yaml`.
- `overlay_manifests` — `label_list(allow_files = [".yaml", ".yml"])`.
- `kustomization` — `label(allow_single_file = True, mandatory = True)`.
  The overlay's top-level `kustomization.yaml`. The directory containing
  this file is what gets passed to `kustomize build`.
- `image_refs` — `string_dict` (see 05).
- `out` — `output` file, defaults to `<name>.yaml`.
- `env_flag`, `tag_flag`, `registry_flag` — `label` attrs with no defaults.
  Callers (or the macro wrapper) pass the flags generated by
  `farakov_lifecycle_environments`. The ruleset has NO opinion on where
  those flags live.
- `_rewriter` — the py_binary from task 05, `cfg = "exec"`.

Impl:
1. Compute the kustomize "source dir" as the dirname of
   `ctx.file.kustomization.short_path` relative to `ctx.bin_dir` — actually,
   the cleanest approach: stage every `base_manifests` + `overlay_manifests`
   + `kustomization` file into a TreeArtifact via
   `ctx.actions.declare_directory`, rooted at a layout dictated by each
   file's `owner.package` (so relative `../../base` refs in
   `kustomization.yaml` still resolve). Then `kustomize build` runs against
   the staged dir.
2. Use the `rules_kustomize` toolchain to locate the `kustomize` binary via
   `ctx.toolchains["@rules_kustomize//kustomize:toolchain_type"]`. No
   `use_default_shell_env`.
3. Run `kustomize build <staged_overlay_dir> --load-restrictor
   LoadRestrictionsNone` and pipe stdout to a temp file.
4. Run the rewriter (task 05 helper) against the temp file → `ctx.outputs.out`.
5. Return `[DefaultInfo(files = depset([out])),
   LifecycleManifestsInfo(files = depset([out]), directories = depset(),
   source_package = ctx.label.package)]`.

Provider requirements:
- `env_flag`, `tag_flag`, `registry_flag` must carry `LifecycleEnvironmentInfo`,
  `LifecycleImageTagInfo`, `LifecycleImageRegistryInfo` respectively (declared
  via `providers = [...]`).

### Macro: `kustomize_manifest`
Ergonomic defaults so callers who follow the `k8s/base`, `k8s/overlays/<env>`
convention can still use a one-liner **without** the ruleset knowing anything
about the convention.

Signature:
```
kustomize_manifest(
    name,
    kustomization,                  # label, required
    base_manifests = None,          # label_list, default []
    overlay_manifests = None,       # label_list, default [kustomization]
    image_refs = None,
    env_flag, tag_flag, registry_flag,   # keyword-required; no defaults
    out = None,
    visibility = None,
)
```
Note the **absence** of hardcoded `//build_flags:*` labels — callers must pass
flag labels explicitly (or wrap this macro in a project-local macro that
passes their generated flags).

## Acceptance criteria
- No reference to `k8s/overlays/<env>` or any env name in this file.
- No `native.glob` inside the rule or macro.
- No `use_default_shell_env`.
- `kustomize` binary is sourced from a toolchain.
- Rule works with overlays placed at arbitrary repo paths (e.g., `deploy/kustomize/my-svc/prod/`).
